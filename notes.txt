There are many search libraries - Apache Lucene, Elasticsearch, Solr; but these are somewhat heavy - example, to run Elasticsearch, you need to start a Docker container. Hence, for small
dataset, we will use lightweight in-memory Python search engine called 'minsearch'. The text fields, keyword fields, boosting and filtering implemented in minsearch are ElasticSearch concepts.
The index tokenizes the text fields and treats keyword fields as exact strings.

Text fields are the fields you search through. When you type a query, the search engine looks at these fields and tokenizes them. It splits text into words, lowercases them, removes stop words, 
and ranks the results by relevance. So question, section, and answer are text fields.

Keyword fields are for exact matching. Think of a SQL query like SELECT * FROM index WHERE course = 'data-engineering-zoomcamp'. The results must come from the specified course, no matter what 
ranking or boosting you do for text fields. You use keyword fields to restrict the search space to a particular subset.

Explain filtering and boosting.

search_results = index.search(
    query=question,
    boost_dict={'question': 2.0, 'section': 0.5},
    filter_dict={'course': 'llm-zoomcamp'},
    num_results=5
)

Filtering - is a binary yes/no gate; a document either matches the filter or it doesn't. This is done to restrict the search to a specific space. Filtering does not affect relevance scores.
It has hard-exclusion property which means that the documents outside the search space would never appear in the result, regardless of how relevant they may be. In ElasticSearch, filtering
improves the search performance as it can cache the results of the filter conditions and reuse them across searches.

Boosting - multiplies the relevance score of a field's match by a given factor, influencing ranking without excluding anything. Not all fields are equally important - example, the 'question' field
in an faqs dataset is usually more relevant than 'section' for matching. Hence, the query words appearing in the question is a stronger signal than them appearing in the section name. 
The weightage attached to different fields reflect this importance. All fields have a default boost of 1. Giving 'question' a boost of 2 means it counts two times as much. Similarly, giving 
'section' 0.5 means it counts half as much, since a match there tells us less.

Say your query is "reset password" and you have two documents:
    Doc         |                   question field                  |               section field
    A           |       "How to reset password" (strong match)      |            "General" (no match)
    B           |              "Account settings" (no match)        |      "Reset Password Section" (strong match)

Score of a doc would be calculated as: (number of query tokens in question field * question_boosting) + (number of query tokens in section field * section_boosting)

Let's take question = 1.0 and section = 1.0 (without boosting)
Score of Doc A = 2 * 1.0 + 0 * 1.0 = 2
Score of Doc B = 0 * 1.0 + 2 * 1.0 = 2
Hence, document A and document B are equally relevant

Let's take question = 2.0 and section = 0.5 (with boosting)
Score of Doc A = 2 * 2.0 + 0 * 0.5 = 4
Score of Doc B = 0 * 2.0 + 2 * 0.5 = 1
Hence, document A is more relevant to answer the query.

A prompt has three parts in an AI system: system instructions, retrieved context and user query. The other optional part is examples (for few-shot prompting). Sometimes, instructions include the output
constraints and formatting instructions as well. Other times, they are considered as a different part. Instructions (system prompt) tells the LLM how to behave - it never changes and remains 
same for every request. The user prompt (retrieved context + user query) changes with every request - it carries the actual question and the retrieved context.

The context is a formatted string with all the search results. It's a small preprocessing step, which creates a format that makes it easy for the llm to read.

--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Tokens

1) Input tokens are the tokens the model reads when processing a request. The include:
    a) The prompt (user query and context)
    b) Any previous conversation history sent to the model
    c) System instructions and tool outputs

2) Cached tokens are input tokens that the model has already processed in a previous request and can reuse from a cache instead of recomputing. Example:
    Turn 1: Tell me about photosynthesis.
    The model processed 2000 tokens of conversation context
    Turn 2: Now explain it for a 10-year old.
    If the first 1,900 tokens are identical to the previous request, those may be counted as cached tokens. The system reuses previous computations for those tokens, reducing cost and latency.

    Cached tokens are still a part of the input context. These are the tokens not generated by the model and are often billed differently.

3) Output tokens are the tokens the model generates in its response.

What is the difference between OpenAI system and developer roles?

In shaping model behavior, OpenAI adhere to the following principles:
1) Maximize users' autonomy and ability to use and customize the tool according to their needs
2) Minimizing harm
3) Choosing sensible defaults (platform-level rules as well as user- and guideline-level defaults, where the latter can be overridden by users or developers)

While the overarching goals provide a directional sense of desired behavior, they are too broad to dictate specific actions in complex scenarios where the goals might conflict. 
For example, how should the assistant respond when a user requests help in harming another person? Maximizing helpfulness would suggest supporting the user's request, but this directly 
conflicts with the principle of minimizing harm.

Each instruction has a level of authority. Instructions with higher authority override those with lower authority. This chain of command is designed to maximize steerability and control 
for users and developers, enabling them to adjust the model's behavior to their needs while staying within clear boundaries:
Platform -> Developer -> User -> Guideline
1) Platform: rules that cannot be overridden by developers or users
2) Developer: instructions given by developers using the API; models should obey developer instructions unless overriden by platform instructions
3) User: instructions from the end users; models should honor user requests unless they conflict with developer- or platform-level instructions

The developer role is the modern standard for providing foundational instructions in reasoning models (like o-series), while the older system role is used to set behavior, constraints, 
tone, goals, and examples of correct responses in non-reasoning models (like GPT-4o). Models like OpenAI's o1 require the developer role to establish core instructions and prevent the 
user from overriding system constraints. Older models convert developer messages to system messages and vice-versa depending on the specific API endpoint.

A practical rule is:
a) Use system for the assistant's overall role and behavior.
b) Use developer for your application's rules and formatting requirements.
c) Use user for the actual request.
Example:
[
  {
    "role": "system",
    "content": "You are a financial education assistant."
  },
  {
    "role": "developer",
    "content": "Always respond using markdown tables when comparing investments."
  },
  {
    "role": "user",
    "content": "Compare index funds and bonds."
  }
]

Though in many simple cases, system and developer roles seem to work the same, the difference becomes important when:
1) Multiple instructions layer exist
2) There are conflicts
3) Tool used is involved

For typical systems that only need “ground rules once”, then instructions do fine, set them there and forget about it. However, some more sophisticated systems need a far more nuanced 
approach, the rules change mid conversations (example, to “inject” system prompts mid agentic stream):

dev message: you are communicating to user on telegram, avoid markdown tables
user: turn X into a table so I can see why it is faster
assistant: replies without a table
dev message: you are communicating to user on web, use rich markdown
user: try again
assistant: lovely table rendered

Minsearch is a lightweight in-memory search library. This means that we have to do document re-indexing everytime we start the notebook or the application as the index isn't persistent. This
approach is fine for small datasets. However, this process breaks down as the dataset grows. Fetching data takes time - calling APIs, parsing files, cleaning text. With millions of documents, 
the startup becomes slow. You don't want to wait minutes every time your service restarts. Hence, we separate ingestion from querying. Ingestion process will write the data to a persistent
search index and another process reads from it. The two processes run independently and only share the index between them. You can use any persistent search backend for this, such as 
Elasticsearch, OpenSearch, or Qdrant. (a custom made library sqlitesearch has the same API as minsearch and uses SQLite FTS5 (fill text search as the backend))

Agentic RAG

In a RAG, the flow is fixed, i.e., search -> build prompt with context -> request to llm. If we use lexical-search and the user's query (exactly) matches the documents, the search finds 
the right documents and a helpful answer is returned. However, often the search returns nothing useful because:
1) the user made a typo
2) the query is asked in an unusual way
3) the information is needed from two different searches
In such cases, there is nothing in our pipeline that recovers from a failed search - the search runs and if it returns garbage, LLM gets garbage. 
Example: If we have this question - "How do I run Ollama locally?" - the search works fine; it finds the relevant documents from faqs and the LLM gives a good answer
However, if we try something like this - "How do I run Olama locally?" - the word 'Ollama' doesn't match the word 'Olama' in our index because we used lexical search which looks for the
exact word and finds nothing. This is the limitation of the fixed pipeline. The search runs once with the exact query the user typed, and there's no second chance. The pipeline doesn't know 
the search failed, so it can't try again with a corrected query.

So, instead of routing the user question straight to search, we can hand control to the LLM and let it drive. The LLM is in charge now can fix typos, search again with different terms, 
and ask the user a clarifying question. A fixed flow can't do any of this. Once we put the LLM in control, our system becomes agentic, so it's flexible rather than rigid. An agent uses an 
LLM to decide which actions to take and in which order. Instead of a fixed flow, the LLM chooses what to do at each step. Example:

User: How do I run Olama? -> LLM: I'll search for 'Olama' -> search() - No useful results -> LLM: Hmm, no results - maybe a typo for 'Ollama' -> search() - Ollama - found results -> LLM: Response

In agentic RAG (Retrieval-Augmented Generation), function calling is the mechanism that lets the LLM decide when to invoke external tools instead of relying only on its own knowledge.
Scenario - What the LLM does without any tool:

messages = [
  {'role':'user', 'content':'I just discovered the course. Can I join it?'}
]

response = openai_client.responses.create(
  model='gpt-5.4-mini',
  input=messages
)

The model answers from its general knowledge, something like "it depends on the course" or "check the course website". It doesn't know about our FAQ, so the answer is vague and not helpful.
This is exactly why we need RAG, and why we want to hand the model a tool.

Define the tool: Define a top-level search function that queries the index directly - the model will reference it by this name (same as the search function we had for RAG)
Now tell the model about this function. The model doesn't see our Python code, only a schema describing what the function does and what arguments it takes. LLMs are language agnostic. 
At the end we're just making an HTTP call, so we describe the tool in JSON rather than in Python. The same schema would work from TypeScript or Java.

***Step 1***
def search(query):
    boost_dict = {"question": 3.0, "section": 0.5}
    filter_dict = {"course": "llm-zoomcamp"}

    return index.search(
        query,
        num_results=5,
        boost_dict=boost_dict,
        filter_dict=filter_dict
    )

search_tool = {
    "type": "function",
    "name": "search",
    "description": "Search the FAQ database for entries matching the given query.", # most important field as the models reads it to decide when to call the function
    "parameters": { # parameters is a JSON schema for the arguments 
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "Search query text to look up in the course FAQ."
            }
        },
        "required": ["query"], # mark 'query' as required so the model always fill it in
        "additionalProperties": False
    }
}

***Step 2***
Now, we send the same question as before, but this time we include the tool in the request:

response = openai_client.responses.create(
    model="gpt-5.4-mini",
    input=messages,
    tools=[search_tool],
)

response.output

***Step 3***
Instead of a message with the answer, the response contains a function_call entry. The model decided it needs to search the FAQ before answering. Rather than reply, it asked us 
to run the search function first. Look at the arguments too. The model didn't pass our question verbatim. It judged the raw question wasn't the best query to search with. 
So it rewrote our question into search keywords like "enroll late join course".

The function call contains JSON arguments. We parse them, call our search function, and serialize the result.

import json

call = response.output[0]
args = json.loads(call.arguments)

results = search(**args)
result_json = json.dumps(results, indent=2)

***Step 4***
Now we send this result back to the model. First, we add the model's output to the conversation history - the model needs to see its own function call. Then we add the tool result.

messages.extend(response.output) # model sees its own function call

messages.append({
    "type": "function_call_output",
    # call_id links the tool result to the specific function call the model requested; if the model makes multiple function calls in one turn, each one gets its own call_id
    "call_id": call.call_id,
    "output": result_json,
})

***Step 5***
We call the API a second time with the expanded history:

response = openai_client.responses.create(
    model="gpt-5.4-mini",
    input=messages,
    tools=[search_tool],
)

response.output_text

This time the model has the original question, its own decision to call search, and the FAQ results. It can now produce a proper course-specific answer. We have to send the 
whole history because LLMs are stateless between API calls. The memory is the list you send as input. If you send only the tool result, the model has no idea what's going on. 
So on this second call we replay everything we have so far. That means the question, the decision to call search, and the result we got back. That's the full function-calling 
loop for a single turn. With plain RAG we made one call, and here we make two. Turning RAG agentic means more round-trips.

Vector Search

The keyword search (text search or lexical search) matches the exact works. Example, if you search for "Docker", the document has to contain "Docker" to be fetched and returned.
But look at these two questions:
a) "Can I still join the course after the start date?"
b) "Is it possible to enroll late?"
They mean the same thing, yet they share almost no words. A keyword search engine struggles to match them. We need something that works on semantic meaning, not on the exact words.
That something is vector search. Instead of matching words, it matches ideas.

We run vector search in two stages.
1) Offline (indexing): we convert all documents into vectors and store them in an index.
2) Online (querying): we convert the user's query into a vector with the same model, then find the closest document vectors by similarity.

An embedding model produces these vectors. It's a neural network trained to capture meaning, so texts that mean similar things land on similar vectors. We measure how close two 
vectors are with a distance metric. The most common one is cosine similarity. Cosine similarity measures the angle between two vectors:
a) Vectors pointing in the same direction: similarity close to 1 (similar)
b) Vectors at right angles (orthogonal): similarity close to 0 (unrelated)
c) Vectors pointing in opposite directions: similarity close to -1 (opposite meaning)

The larger the cosine similarity, the more similar the two texts are in meaning. Because our vectors are normalized, the dot product gives us cosine similarity directly. 
This is why we can use dot product to compare texts. In practice, we rarely get cosine similarity below 0. The embedding model maps text to a region of the vector space where most 
vectors have positive components. There's no concept of "opposite meaning" that maps to a vector pointing the other way.

Differences between keyword search and vector search:
a) Keyword search matches exact words; Vector search matches meaning
b) Keyword search suits specific terms, IDs, and names; Vector search suits paraphrased questions and natural language
c) Keyword search uses an inverted index (BM25, TF-IDF); Vector search uses a vector index based on cosine similarity.
d) Keyword search misses synonyms and paraphrases; Vector search misses exact term matches.

Vector search is usually better, but it adds a lot of operational complexity. A good idea is to start with keyword search and reach for vector search once you can show they are
woth the extra cost (or if you have a GPU!. In practice, the two work best together. Hybrid search combins them and usually is considered a best practice.

--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

The embedding model learns to place words as points in a multi-dimensional space - each dimension represents a concept the model learned. Words with similar meanings land close to each 
other (Word2Vec). The same idea works for entire sentences. Example:
Q1: "I just discovered the course. Can I still join it?"
Q2: "I just found out about the program. Can I still enroll?" - The vectors for Q1 and Q2 are close as they mean the same thing.
Q3: "How do I run Docker on Windows?" - This one is far away from Q1 and Q2.

Each of the documents in our dataset becomes a point in this space. An embedding model takes text in and returns a fixed-length array of numbers. When a user asks a question, we embed the 
query into the same space and find the closest documents (nearest documents) which become our search results. Since the model encodes the whole sentence, not the words in isolation. 
So it can tell apart the same word in different contexts. Example: "Apple is a great company" and "An apple a day keeps the doctor away" both contain the word "apple". However, 
the surrounding context changes the enbedding.
