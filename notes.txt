There are many search libraries - Apache Lucene, Elasticsearch, Solr; but these are somewhat heavy - example, to run Elasticsearch, you need to start a Docker container. Hence, for small
dataset, we will use lightweight in-memory Python search engine called 'minsearch'. The text fields, keyword fields, boosting and filtering implemented in minsearch are ElasticSearch concepts.
The index tokenizes the text fields and treats keyword fields as exact strings.

Text fields are the fields you search through. When you type a query, the search engine looks at these fields and tokenizes them. It splits text into words, lowercases them, removes stop words, 
and ranks the results by relevance. So question, section, and answer are text fields.

Keyword fields are for exact matching. Think of a SQL query like SELECT * FROM index WHERE course = 'data-engineering-zoomcamp'. The results must come from the specified course, no matter what 
ranking or boosting you do for text fields. You use keyword fields to restrict the search space to a particular subset.

Explain filtering and boosting.

search_results = index.search(
    query=question,
    boost_dict={'question': 2.0, 'section': 0.5},
    filter_dict={'course': 'llm-zoomcamp'},
    num_results=5
)

Filtering - is a binary yes/no gate; a document either matches the filter or it doesn't. This is done to restrict the search to a specific space. Filtering does not affect relevance scores.
It has hard-exclusion property which means that the documents outside the search space would never appear in the result, regardless of how relevant they may be. In ElasticSearch, filtering
improves the search performance as it can cache the results of the filter conditions and reuse them across searches.

Boosting - multiplies the relevance score of a field's match by a given factor, influencing ranking without excluding anything. Not all fields are equally important - example, the 'question' field
in an faqs dataset is usually more relevant than 'section' for matching. Hence, the query words appearing in the question is a stronger signal than them appearing in the section name. 
The weightage attached to different fields reflect this importance. All fields have a default boost of 1. Giving 'question' a boost of 2 means it counts two times as much. Similarly, giving 
'section' 0.5 means it counts half as much, since a match there tells us less.

Say your query is "reset password" and you have two documents:
    Doc         |                   question field                  |               section field
    A           |       "How to reset password" (strong match)      |            "General" (no match)
    B           |              "Account settings" (no match)        |      "Reset Password Section" (strong match)

Score of a doc would be calculated as: (number of query tokens in question field * question_boosting) + (number of query tokens in section field * section_boosting)

Let's take question = 1.0 and section = 1.0 (without boosting)
Score of Doc A = 2 * 1.0 + 0 * 1.0 = 2
Score of Doc B = 0 * 1.0 + 2 * 1.0 = 2
Hence, document A and document B are equally relevant

Let's take question = 2.0 and section = 0.5 (with boosting)
Score of Doc A = 2 * 2.0 + 0 * 0.5 = 4
Score of Doc B = 0 * 2.0 + 2 * 0.5 = 1
Hence, document A is more relevant to answer the query.

A prompt has three parts in an AI system: system instructions, retrieved context and user query. The other optional part is examples (for few-shot prompting). Sometimes, instructions include the output
constraints and formatting instructions as well. Other times, they are considered as a different part. Instructions (system prompt) tells the LLM how to behave - it never changes and remains 
same for every request. The user prompt (retrieved context + user query) changes with every request - it carries the actual question and the retrieved context.

The context is a formatted string with all the search results. It's a small preprocessing step, which creates a format that makes it easy for the llm to read.

Tokens

1) Input tokens are the tokens the model reads when processing a request. The include:
    a) The prompt (user query and context)
    b) Any previous conversation history sent to the model
    c) System instructions and tool outputs

2) Cached tokens are input tokens that the model has already processed in a previous request and can reuse from a cache instead of recomputing. Example:
    Turn 1: Tell me about photosynthesis.
    The model processed 2000 tokens of conversation context
    Turn 2: Now explain it for a 10-year old.
    If the first 1,900 tokens are identical to the previous request, those may be counted as cached tokens. The system reuses previous computations for those tokens, reducing cost and latency.

    Cached tokens are still a part of the input context. These are the tokens not generated by the model and are often billed differently.

3) Output tokens are the tokens the model generates in its response.

What is the difference between OpenAI system and developer roles?

In shaping model behavior, OpenAI adhere to the following principles:
1) Maximize users' autonomy and ability to use and customize the tool according to their needs
2) Minimizing harm
3) Choosing sensible defaults (platform-level rules as well as user- and guideline-level defaults, where the latter can be overridden by users or developers)

While the overarching goals provide a directional sense of desired behavior, they are too broad to dictate specific actions in complex scenarios where the goals might conflict. 
For example, how should the assistant respond when a user requests help in harming another person? Maximizing helpfulness would suggest supporting the user's request, but this directly 
conflicts with the principle of minimizing harm.

Each instruction has a level of authority. Instructions with higher authority override those with lower authority. This chain of command is designed to maximize steerability and control 
for users and developers, enabling them to adjust the model's behavior to their needs while staying within clear boundaries:
Platform -> Developer -> User -> Guideline
1) Platform: rules that cannot be overridden by developers or users
2) Developer: instructions given by developers using the API; models should obey developer instructions unless overriden by platform instructions
3) User: instructions from the end users; models should honor user requests unless they conflict with developer- or platform-level instructions

The developer role is the modern standard for providing foundational instructions in reasoning models (like o-series), while the older system role is used to set behavior, constraints, 
tone, goals, and examples of correct responses in non-reasoning models (like GPT-4o). Models like OpenAI's o1 require the developer role to establish core instructions and prevent the 
user from overriding system constraints. Older models convert developer messages to system messages and vice-versa depending on the specific API endpoint.

A practical rule is:
a) Use system for the assistant's overall role and behavior.
b) Use developer for your application's rules and formatting requirements.
c) Use user for the actual request.
Example:
[
  {
    "role": "system",
    "content": "You are a financial education assistant."
  },
  {
    "role": "developer",
    "content": "Always respond using markdown tables when comparing investments."
  },
  {
    "role": "user",
    "content": "Compare index funds and bonds."
  }
]

Though in many simple cases, system and developer roles seem to work the same, the difference becomes important when:
1) Multiple instructions layer exist
2) There are conflicts
3) Tool used is involved

For typical systems that only need “ground rules once”, then instructions do fine, set them there and forget about it. However, some more sophisticated systems need a far more nuanced 
approach, the rules change mid conversations (example, to “inject” system prompts mid agentic stream):

dev message: you are communicating to user on telegram, avoid markdown tables
user: turn X into a table so I can see why it is faster
assistant: replies without a table
dev message: you are communicating to user on web, use rich markdown
user: try again
assistant: lovely table rendered

Minsearch is a lightweight in-memory search library. This means that we have to do document re-indexing everytime we start the notebook or the application as the index isn't persistent. This
approach is fine for small datasets. However, this process breaks down as the dataset grows. Fetching data takes time - calling APIs, parsing files, cleaning text. With millions of documents, 
the startup becomes slow. You don't want to wait minutes every time your service restarts. Hence, we separate ingestion from querying. Ingestion process will write the data to a persistent
search index and another process reads from it. The two processes run independently and only share the index between them. You can use any persistent search backend for this, such as 
Elasticsearch, OpenSearch, or Qdrant. (a custom made library sqlitesearch has the same API as minsearch and uses SQLite FTS5 (fill text search as the backend))
